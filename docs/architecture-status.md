# Architecture status

> **CURRENT — implementation checkpoint:** through commit `6ce7a22` on `main`,
> verified 2026-07-12. Later documentation-only edits do not change the runtime
> described here.

This register uses five labels:

- **CURRENT** — directly present in committed code or repository structure.
- **DECIDED** — selected direction, whether or not implemented.
- **PROPOSED** — candidate requiring review before implementation.
- **DEPRECATED** — behavior retained temporarily but intended for removal.
- **UNKNOWN** — repository evidence is insufficient or contradictory.

An accepted decision is not evidence that its implementation exists. In
particular, the presence of a TypeScript interface is not evidence of a route,
database collection, validator, user workflow, or export.

| Topic | Status | Current implementation | Intended direction | Risk if changed now | Needed decision |
|---|---|---|---|---|---|
| Frontend stack | **DECIDED** | No `src/app/`, HTML entry point, service worker, or runnable field client exists. `tsconfig.json` currently compiles all `src/**/*.ts` into `pb_public/assets`. | Framework-free, mobile-first TypeScript and static assets under `src/app/`, generated into `pb_public/`, with no runtime CDN. | A framework or incompatible build pipeline would violate the field/offline and repository constraints before the first UI exists. | Define the minimum source-to-static build contract and distinguish domain output from deployable app assets. |
| Backend and persistence | **DECIDED** | No PocketBase migration, hook, collection, repository, database, or server route is committed. | One LineCheck PocketBase process with one operator-owned SQLite database and file storage. | Premature schema work would freeze incomplete response validation, segment derivation, authorization, and ID decisions. | Finish A-001/A-002 and settle migration-facing authorization/public-ID choices before the first migration. |
| Public identifier strategy | **PROPOSED** | `EntityId` is a string and `createEntityId()` creates a 15-character PocketBase-compatible ID. There is no separate cross-instance public ID. | ADR 0006 proposes immutable, globally collision-resistant public IDs distinct from local database IDs. | Publishing PocketBase IDs now could make restores, imports, and cross-product references storage-dependent. | Accept, amend, or reject ADR 0006 and define backfill/collision behavior. |
| Offline storage | **UNKNOWN** | No `src/app/`, IndexedDB layer, service worker, local draft store, attachment queue, or storage policy exists. | **DECIDED:** unlocked drafts and pending evidence must survive an interruption and show honest unsynced/stale state. The storage mechanism remains **UNKNOWN**. | Choosing storage before payload and upgrade rules settle can strand drafts or sensitive signature/evidence data. | Select browser storage boundaries, quota/retention rules, encryption expectations, and upgrade behavior. |
| Synchronization | **DECIDED** | `SyncOperation` is a discriminated union with strict runtime payload validation for every permitted entity/operation pair; signatures and child updates are excluded. There is no outbox, sync route, receipt store, or conflict resolver. | Idempotent mutations keyed by client and `mutation_id`; append-oriented children; visible conflicts; no merge of signed or locked evidence. | Treating validated DTOs as a working protocol could still create duplicate evidence because persistence and receipts do not exist. | Define receipt lifetime, batch semantics, route authorization, revision conflicts, and conflict UX. |
| Append-only facts | **DECIDED** | Pure audit append/hash helpers and aggregate mutability guards exist. No persistence rule prevents update or delete. | Executed readings, attachments, signatures, snapshots, void records, and audit events are append-oriented; ordinary state is not event sourced. | UI-only enforcement would allow API or administrator workflows to rewrite evidence silently. | Define the exact draft-to-executed boundary and enforce it in migrations/routes with tests. |
| Derived status | **DECIDED, PARTIAL** | `TestSegment.status` remains a response field and the legacy transition helper remains, but strict create/update inputs reject `status`; signed/void values were removed from the segment enum. | Derive segment acceptance/standing from authoritative test and replacement facts. | Retaining the transition helper without a derivation fold can still create a second status authority. | A-002 must replace the stored-state transition authority with a tested derivation including signed pass/fail and void/replacement chains. |
| Calculation method | **DECIDED, PARTIAL** | Exact decimal arithmetic, conversions, makeup-water summation, and `project_specified_allowance@1.0.0` exist. Template/request/result types and runtime request validation now accept only `fixture_only` and retain source/time provenance. | MVP accepts only the clearly labeled project-specified allowance fixture and never derives or adopts a universal engineering formula. | Cross-unit requests, full golden boundaries, and an authoritative server recomputation path are still absent. | Complete A-005 conversion/provenance vectors and define a later project-authorized method separately. |
| Signature capture | **DECIDED** | Signature, witness-attestation, and snapshot interfaces exist. No UI, protected storage, route, transaction, identity proofing, or validation exists. | Accountless witness review of an allowlisted frozen record, with signer identity/role/company, exact attestation, time, optional consented location, and protected signature evidence. | Capturing a signature before atomic validation could present a partial or stale record as final. | Define witness authorization/session handling, signature file limits, consent, retention, and failure recovery. |
| Record locking | **DECIDED** | `assertAggregateMutable()` rejects an aggregate with `locked_at`, a signed snapshot, or a void record. No database or route enforces the guard. | One authoritative atomic sign-and-lock action; ordinary post-lock writes fail; corrections preserve the original through void-and-replace. | A non-atomic implementation can leave a signature without a matching frozen record or a partially locked aggregate. | Define transaction order, idempotent retry result, authorization, and void/replacement release cut. |
| Evidence hashing | **DECIDED** | Generic NFC/sorted-key canonicalization and UTF-8 SHA-256 helpers exist under `src/domain/`; snapshot types exist. No allowlisted snapshot builder, golden byte vector, or stored hash exists. | Hash the exact bytes of a versioned canonical signed snapshot; do not hash report/PDF presentation bytes or claim legal/absolute immutability. | Ratifying the generic serializer without field and ordering vectors could freeze ambiguous signing bytes. | Review `linecheck-c14n-v1`, array ordering/ties, numeric/timestamp normalization, module ownership, and golden vectors. |
| PDF generation | **DECIDED** | No report assembler, print HTML, browser visual test, PDF renderer, or generated PDF exists. Type/event vocabulary mentioning PDF is not a feature. | First deliver deterministic print-ready HTML. A production PDF renderer remains later work and core capability when actually tested. | Selecting a renderer now may add a second service or make presentation bytes part of the evidence contract. | Define the HTML report specification first; defer renderer selection until its content and visual acceptance tests exist. |
| API versioning | **DECIDED, IN PROGRESS** | `src/contracts.ts` contains explicit v1 response identifiers, strict create/update DTOs, foundational response parsers, a safe versioned `ApiError`, and compatibility tests proving additive project fields are stripped. Remaining response parsers and HTTP routes are absent. | Versioned DTOs rather than raw PocketBase records; additive response fields tolerated; mutation fields allowlisted; breaking semantics require a new version or migration plan. | Publishing the partial parser set as a complete API would leave some consumers unprotected. | Finish A-001 response parsers/vectors, then publish routes only after persistence and authorization exist. |
| Cross-product handoffs | **DECIDED** | No LineCheck import, export, lifecycle event, manifest, consumer, or producer exists. | Separate authorities exchange versioned exports/manifests/events with provenance; no shared database or mandatory broker. Draft shapes are **PROPOSED** in `ecosystem-contracts.md`. | Implementing one-off mappings before IDs and ownership settle can create a second unversioned contract. | Validate the draft with TrenchNote, MainLine, LoopCheck, and future sidecar consumers using fictional fixtures. |
| Paid/core boundary | **DECIDED** | The boundary is documented; `linecheck-lookahead` is an empty local sibling. No integration exists. | Core retains field capture, evidence, signing, locking, self-hosting/backups, and basic report/data exit; paid sidecars add optional operations and aggregation through public contracts. | Moving a required field capability to a paid component would make core incomplete and break offline/self-host use. | Define “basic export/report” release criteria and sidecar credentials only when a consumer exists. |
| Service-cutover ownership | **PROPOSED** | LineCheck has no service model. LoopCheck currently implements service records, six cutover phases, service import/pages, and a cutover board. | ADR 0007 proposes eventual LineCheck ownership after its pressure-test foundation and migration contracts exist; no dual write or immediate move. | Moving now would abandon working history; duplicating creates competing authorities and PII risk. | Cross-repository owner decision, field validation, versioned mapping, privacy review, and per-project transition plan. |
| MainLine relationship | **UNKNOWN** | MainLine is a discovered sibling for station-based pothole, alignment, joint-construction, and joint-test records. LineCheck has no integration or documented mapping to MainLine stations/joints. | Keep MainLine authoritative for constructed-location/joint facts and LineCheck authoritative for its acceptance records; exchange references only where a real workflow needs them. | Treating a station, joint, and test segment as the same entity would erase different scopes and correction rules. | Validate segment-to-alignment/station reference semantics and whether any lifecycle handoff is needed. |
| Deployment topology | **DECIDED** | No PocketBase binary, container, service unit, deployment script, backup command, or restore test is committed. `.env.example` only records placeholders. | One PocketBase/SQLite process serving the static app on one machine or in one container, with no required cloud/queue/second database. | Deployment automation before schema, authorization, and backup contents settle can create a false production path. | Choose supported operating shapes, TLS/reverse-proxy guidance, backup consistency, and restore verification before release. |
| Authentication | **UNKNOWN** | No auth collection, rule, login UI, token handling, role model, or route authorization exists. `SECURITY.md` requires closed defaults; MVP prose also wants accountless witness attestation. | Least-privilege operators and integrations, no public self-registration by default, protected evidence, and a narrowly scoped witness flow. | An ad hoc anonymous signature route or permissive migration could expose PII/evidence or allow forged mutations. | Define actors/roles, witness session boundaries, integration credentials, evidence access, and recovery before the first migration. |

## Current stabilization boundary

**DECIDED:** contract semantics, canonical bytes, authority, and persistence
constraints must stabilize before broad UI or schema implementation.

**CURRENT:** unit tests now exercise the foundational contract/domain surface.
No item requiring persistence, browser behavior, authorization, signing, or
exports is production-ready until its integration or end-to-end criteria pass.
