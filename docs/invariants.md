# Invariants

This document separates rules enforced by committed pure code from product
rules that are decided but not yet enforced and ecosystem rules that remain
proposed. A TypeScript type or helper is not a persistence, authorization, or
workflow guarantee.

## CURRENT code-level rules

These rules are present at commit `34693b4`. They have no committed tests, so
they are **CURRENT implementation**, not a verified release claim.

| CURRENT rule | Evidence | Limit |
|---|---|---|
| Entity IDs generated by the domain helper are 15 lowercase alphanumeric characters and are PocketBase-compatible. | `src/domain/ids.ts` | This is not a distinct stable public-ID scheme and does not validate every string typed as `EntityId`. |
| Plain decimal strings exclude exponent notation, enforce at most 18 integer digits and 12 fractional digits, and use exact `bigint` arithmetic. | `src/domain/decimal.ts` | Callers must select the non-negative parser where negative measurements are invalid. Runtime DTO validators do not exist. |
| Decimal display rounding is half-up; unit conversion uses a versioned half-up scale-12 policy. | `src/domain/decimal.ts`, `src/domain/units.ts` | Conversion precision has no golden tests. Diameter conversion is not implemented as a separate helper. |
| The allowance fixture accepts only method `project_specified_allowance@1.0.0`, requires equal input volume units, and passes when actual makeup water is at most allowance plus tolerance. | `src/domain/calculation.ts` | The wrapper currently permits a template marked `project_approved`; this conflicts with the MVP fixture-only decision. |
| Makeup-water summation is exact, requires non-negative increments, and rejects mixed units rather than converting them implicitly. | `sumMakeupWater()` in `src/domain/calculation.ts` | No route authoritatively recomputes or persists the total. |

## DECIDED product invariants not yet fully enforced

1. **DECIDED — project authority governs calculations.** No calculation is
   universal. The MVP method compares a project-supplied allowance and must be
   labeled `fixture_only`; it does not derive an allowance or claim adoption of
   an owner, AWWA, or jurisdictional formula.

2. **DECIDED — raw inputs and provenance survive.** Calculation inputs, units,
   method ID/version/status, source reference, rounding/comparison rules,
   unrounded and display results, warnings, and calculation time remain with
   the record. A method change does not silently recalculate history.

3. **DECIDED — units and time meanings are explicit.** Wire measurements use
   contract-defined units; locale or display text never supplies a unit. Wire
   timestamps use RFC 3339 with an offset or `Z`; project calendar dates use
   `YYYY-MM-DD`. The unresolved duration type is listed below.

4. **DECIDED — authoritative boundaries recompute.** A client preview is not
   evidence of pass/fail. The authoritative mutation boundary validates inputs
   and recomputes the fixture result before completion or signing.

5. **DECIDED — executed evidence is append-oriented.** Accepted readings,
   attachments, witness attestations, audit events, and signed snapshots are
   not silently edited or deleted through ordinary operations. Draft setup
   fields may remain mutable under revision and conflict rules.

6. **DECIDED — signing freezes an exact snapshot.** Signing constructs an
   allowlisted, versioned canonical snapshot, hashes its exact UTF-8 bytes with
   SHA-256, stores the snapshot and hash, records the attestation, and locks the
   aggregate atomically. Hashing establishes tamper-evident application
   evidence, not identity proof, trusted time, legal enforceability, or
   protection from a compromised administrator.

7. **DECIDED — corrections preserve history.** A locked record rejects
   ordinary mutation. A correction records actor, time, reason, original hash,
   and a distinct replacement link without changing the original canonical
   bytes. A failed signed test remains valid evidence.

8. **DECIDED — audit is evidence, not the primary state store.** Audit events
   record meaningful operations and may carry a documented previous hash, but
   LineCheck does not reconstruct all operational state through event sourcing.

9. **DECIDED — offline state is honest.** Draft, pending-upload, pending-sign,
   unsynced, stale, conflict, rejected, locked, and export states must be
   distinguishable. Offline staging never presents a signature or lock as
   final before server validation.

10. **DECIDED — project and physical scope are unambiguous.** A pressure test
    belongs to one project and one identified test segment with explicit limits
    and specification context. A human label may aid field recognition but is
    not assumed globally unique.

11. **DECIDED — imported facts retain authority and provenance.** A copied
    project, location, installation reference, or lifecycle fact names its
    producer and source identifier. Importing a fact does not silently transfer
    authority or permit mutation of the source record.

12. **DECIDED — basic field capability remains open.** Capture, fixture
    calculation, evidence, witness attestation, lock, self-hosting/backups, and
    basic report/data exit cannot depend on `linecheck-lookahead` or another
    paid service.

13. **DECIDED — current authorities remain authoritative during migration.**
    TrenchNote remains authoritative for its custody/movement facts, MainLine
    for its station/alignment and constructed-joint facts, and LoopCheck for
    service-cutover records it currently creates. A proposed future boundary
    does not rewrite history or authorize dual write.

## PROPOSED ecosystem invariants

1. **PROPOSED — stable public IDs are separate from local storage IDs.** ADR
   0006 proposes immutable external identifiers for referencable records while
   keeping PocketBase IDs local.

2. **PROPOSED — handoffs use versioned envelopes.** Project references,
   external references, evidence metadata, lifecycle events, manifests, and
   import provenance would use the non-binding shapes in
   [`ecosystem-contracts.md`](ecosystem-contracts.md).

3. **PROPOSED — LineCheck may become the future service-cutover authority.**
   ADR 0007 requires a versioned, evidence-preserving, per-project migration;
   it does not authorize an implementation or move today.

4. **PROPOSED — cross-product status is derived locally from sourced facts.**
   An aggregator may calculate a view, but it does not become authoritative for
   a producer's evidence or write its derived state back into the producer.

## UNKNOWN or contradictory rules blocking contract freeze

| Topic | Conflicting evidence | Required resolution |
|---|---|---|
| Segment status | `TestSegment.status` and a transition graph are CURRENT; MVP/backlog prose says status is derived. | Decide stored state versus derived response, client input rules, and the void/replacement fold. |
| Void semantics | `PressureTestResult` includes `void`; DECIDED documentation treats void as a separate disposition. | Remove/separate or explicitly migrate the v1 shape before validators and persistence. |
| Duration | Repository rules call durations integer minutes; pressure-test and template DTOs use `DecimalString`. | Select an integer representation and compatibility plan. |
| Fixture approval | The wrapper accepts `project_approved`; the MVP permits only `fixture_only`. | Add an MVP guard and defer a real authorization mechanism. |
| Canonical array order | The generic serializer preserves input order; DECIDED signing requires deterministic reading/evidence ordering and tie handling. | Specify builder sorting, duplicate/tie rejection, and golden vectors. |
| API error versioning | `ApiError` exists but has no contract-version field or runtime validator. | Define the versioned envelope and unknown-code behavior before routes ship. |

No row in this final section is an invariant yet. Each belongs in the decision
backlog and must be resolved before dependent migrations, signing, or public
API work.
